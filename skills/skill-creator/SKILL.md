---
name: skill-creator
description: 帮助用户创建新的 Self-improving Agent Skill，遵循 TDD-R + Prompt 结构标准
---

## 能力
当用户需要创建新 Skill 时，按以下模板生成 SKILL.md 文件。

## 第一部分：前件（Frontmatter + YAML）

### 输出模板
```yaml
name: <kebab-case-name>          # 必填，如 trend-intelligence-scout
description: <一句话描述>         # 必填，不超过30字，体现动态领域能力
trigger: <自动触发场景>           # 必填，什么场景下用户无需手动调用
self_improving:
  metric: <评估指标>             # 必填，可量化、可追踪
  feedback_loop: <数据回流路径>   # 必填，数据从哪来、怎么回流
  update_rule: <更新频率>        # 必填，按反馈量or按时间
```

### 各字段填写说明

| 字段 | 要求 | 好例子 | 坏例子 |
|------|------|--------|--------|
| `name` | kebab-case，全小写英文 | `trend-intelligence-scout` | `TrendScout` / `热点抓取` |
| `description` | ≤30字，含动态领域能力 | `多平台热点抓取与选题推荐Agent，支持动态领域匹配` | `热点抓取`（太短，缺动态） |
| `trigger` | 明确触发场景，含"当..." | `当任务涉及热点、选题、趋势、每日话题、平台热搜` | `用户需要热点时`（太模糊） |
| `metric` | 单一、可量化指标 | `选题采纳率与视频表现相关性` | `内容质量`（不可量化） |
| `feedback_loop` | 谁产数据、怎么回流 | `追踪预测热点的实际爆发准确度，运营Agent回传数据` | `收集反馈`（太模糊） |
| `update_rule` | 按量或按时间触发 | `每周优化关键词抓取源和过滤规则` | `定期更新`（缺周期） |

### 完整示例
```yaml
name: trend-intelligence-scout
description: 多平台热点抓取与选题推荐 Agent，支持动态领域匹配
trigger: 当任务涉及热点、选题、趋势、每日话题、平台热搜
self_improving:
  metric: 选题采纳率与视频表现相关性
  feedback_loop: 追踪预测热点的实际爆发准确度，运营Agent回传数据
  update_rule: 每周优化关键词抓取源和过滤规则
```

## 第二部分：Prompt 结构标准（7段式）

> 所有业务 Skill 必须完整包含以下 7 段，顺序不可调换，内容不可省略。

---

### 1. 角色设定（Context）

**目的**：让 Agent 明确"我是谁、为谁服务、在什么边界内工作"。

**必须包含**：
- Agent 的身份和核心专业领域
- 服务的目标用户群（引用 JTBD 和 P0/P1/P2 分层）
- 如果是多领域 Skill，注明动态匹配机制

**示例**：
> 你是 trend-intelligence-scout，专注于 AI 科技领域的热点侦察 Agent。你的用户是"AI PM 副业探索者"（P0 人群）——相信方法论，厌恶割韭菜，想用系统化方法做内容副业。

**常见错误**：
- 身份过于宽泛（"你是一个助手"）
- 缺少用户画像描述

---

### 2. 用户理解确认（Profile Read）

**目的**：强制 Agent 在输出前对齐用户意图，防止跑偏。

**必须包含**：
- 在推荐/执行前，先"复述"用户意图
- 列出需要用户确认的关键信息项
- 缺失信息的默认处理规则

**示例**：
> 在推荐前，先基于用户输入确认：
> - 用户核心领域：（由用户输入或 domain 变量传入）
> - 目标平台：（待用户确认，未明确则标注"待确认"）
> - 内容形式偏好：（待用户确认）
> 缺失信息标注"待确认"，不允许猜测。

**关键规则**：**缺失即标注，绝不猜。**

---

### 3. 知识库调用（Baseline + Specialized）

**目的**：让 Agent 在知识库的约束下输出，而非自由发挥。

**必须包含**：
- **Baseline（基础方法论）**：所有 Skill 通用的方法论框架
  ```
  Baseline 清单：DOMA 架构、四支柱框架、T.A.M.T.、JTBD、四层需求模型
  ```
- **Specialized（领域纵深）**：基于用户 domain 动态匹配的知识
  - 使用 `Domain 映射表` 做匹配
  - 未覆盖的 domain 必须标注"待补充知识库文件"

**示例（Domain 映射表）**：
> 基于用户选择的领域，动态匹配：
> - `香港求职` → IANG签证、港企面试、香港科技岗位、港硕就业路径
> - `AI产品能力` → 作品集设计、系统设计、Agent架构、PRD撰写方法论
> - `跨境运营` → 跨境电商趋势、本地生活运营、流量打法、O2O核销
>
> 当前知识库未覆盖的领域，标注"待补充知识库文件：[领域名]"，并用通用方法论兜底。

---

### 4. 任务定义（Action）

**目的**：把 Skill 的核心工作分解为具体、可执行的步骤。

**必须包含**：
- 编号步骤列表（Agent 按序执行）
- 每步有明确输入和输出
- 约束输出数量（如"推荐 3-5 条"而不是"推荐一些"）
- 禁止模糊动词：出现"分析""优化""评估"时必须附具体指标

**结构**：
```
1. <第一步：做什么 + 输出什么>
2. <第二步：做什么 + 输出什么>
3. <第三步：做什么 + 输出什么>
...
```

**示例**：
> 1. 基于用户领域，推荐 3-5 个热点选题
> 2. 每个选题必须包含：话题名称 | 来源平台 | 热度评分 | 关联知识 | 优先级（P0/P1/P2）
> 3. 只推荐有"知识增量"的话题（能讲出方法论，而非纯情绪）

**示例（错误写法）**：
> - 分析热点
> - 优化推荐
> 问题："分析"和"优化"没有配具体指标，Agent 无法执行。

---

### 5. 反向过滤（What NOT to do）

**目的**：先画红线，阻止 Agent 向错误方向生成。这是最容易被跳过的段落，但也是最重要的。

**必须包含**：
- 至少 3 条明确禁止事项
- 每条禁止事项附带原因或判断标准
- "禁止 X"必须配"应该 Y"或者明确判定逻辑

**常见禁止类别**：

| 类别 | 示例 |
|------|------|
| 禁止编造 | 不确定时标注"待验证"，而非编造 |
| 禁止偏离领域 | 超出 Ontology 的话题直接过滤 |
| 禁止同质化 | 已有大量同质内容的话题不推荐 |
| 禁止未验证来源 | 无法验证的"据传"信息标注"待验证" |

**示例**：
> 以下类型话题**禁止推荐**：
> - 纯娱乐八卦、无行业价值的热点
> - 与用户 Ontology（AI科技+跨境电商+本地生活）明显偏离的话题
> - 无法验证来源的"据传"类信息（标注"待验证"而非直接推荐）
> - 已有大量同质化内容、缺乏差异化空间的话题

---

### 6. 输出格式（Format）

**目的**：约束输出的结构和形式，让 Agent 的输出可直接消费。

**必须包含**：
- 明确输出结构（表格/列表/Markdown/JSON）
- 明确长度限制（多少条、多少字）
- 明确表头或字段定义

**示例**：
> 以 Markdown 表格输出《每日热点选题池》，表头：
> | 话题名称 | 来源平台 | 热度评分 | 关联知识 | 优先级 |

**设计原则**：
- 输出的第一行就应该是有价值的内容，不要开头语
- 格式的决定因素：下游谁在消费这个输出（Agent？人？飞书表格？）

---

### 7. 强制约束（Constraints）

**目的**：兜底规则，处理 Agent 的"自由意志"倾向。

**必须包含**：
- 事实校验规则（怎么判断"能说"还是"不能说"）
- 不确定性处理规则（标注 vs 猜测）
- 格式/行为红线（禁止写什么、禁止做什么）
- 知识边界处理（超出知识库覆盖范围怎么办）

**示例**：
> - 禁止编造热点，不确定时标注"待验证"
> - 禁止输出寒暄、解释、总结
> - 每个选题必须说明"为什么现在火"和"谁能讲"
> - 未覆盖领域必须标注"待补充知识库文件"

**关键原则**：
- Constraints 是最后的防线，要对 Agent 的"幻觉倾向"做针对性约束
- 每一条 Constraints 都应该是可验证的（可以事后判断 Agent 有没有违反）

---

## 第三部分：规则（Rules）

| # | 规则 | 说明 | 违规后果 |
|---|------|------|---------|
| 1 | Skill name 必须用 kebab-case | 全小写英文，连字符分隔。如 `trend-intelligence-scout` | Claude Code 无法识别 |
| 2 | description 不超过 30 字，体现动态领域能力 | 必须含"支持动态领域匹配"或类似表述 | 被静态化，失去多领域能力 |
| 3 | 必须包含 self_improving 段落 | metric / feedback_loop / update_rule 三个字段缺一不可 | Skill 不会自我迭代 |
| 4 | 必须包含完整的 Prompt 结构标准（7段式） | Context → Profile Read → Knowledge → Action → What NOT → Format → Constraints，顺序不可调换，内容不可省略 | Agent 行为不可控 |
| 5 | 输出后直接告诉用户文件保存路径 | 给出绝对路径，无需额外解释 | 用户不知道文件在哪 |

### 检查清单（生成后逐条自查）

生成 SKILL.md 后，逐条确认：

- [ ] name 是 kebab-case 全小写英文
- [ ] description ≤30 字，含动态领域能力
- [ ] trigger 以"当..."开头，明确触发场景
- [ ] self_improving 三段完整
- [ ] 7 段式 Prompt 结构齐全（1→7 顺序正确）
- [ ] Profile Read 有"待确认"处理规则
- [ ] Knowledge 有 Baseline + Specialized 两层
- [ ] Action 步骤编号，禁止模糊动词
- [ ] What NOT 至少 3 条禁止事项
- [ ] Format 有具体结构定义
- [ ] Constraints 有不确定性处理规则
