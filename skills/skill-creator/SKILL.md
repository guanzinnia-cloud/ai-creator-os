# Skill Creator — 高质量Agent Skill 生成器

## 能力

当用户需要创建新 Skill 时，遵循 7 段式 Prompt 结构标准生成 `SKILL.md` 文件。生成完成后自动执行质量评分和效果验证，确保输出达到可部署水平。

---

## 第一部分：前件（Frontmatter + YAML）

### 输出模板

```yaml
name: <kebab-case-name>          # 必填，如 order-reminder-agent
description: <一句话描述>         # 必填，不超过30字，体现动态领域能力
trigger: <自动触发场景>           # 必填，以"当..."开头，明确触发场景
self_improving:
  metric: <评估指标>             # 必填，单一、可量化、可追踪
  feedback_loop: <数据回流路径>   # 必填，数据从哪来、怎么回流
  update_rule: <更新频率>        # 必填，按反馈量或按时间
```

### 各字段填写说明

| 字段 | 要求 | 好例子 | 坏例子 |
|------|------|--------|--------|
| `name` | kebab-case，全小写英文 | `order-reminder-agent` | `OrderReminder` / `订单提醒` |
| `description` | ≤30字，含动态领域能力 | `SAP订单交货期预警与销售提醒Agent，支持多系统数据源` | `提醒Agent`（太短，缺动态）|
| `trigger` | 明确触发场景，含"当..." | `当任务涉及订单管理、交货跟踪、异常预警、SAP系统对接` | `用户需要提醒时`（太模糊）|
| `metric` | 单一、可量化指标 | `预警准确率与人工确认率` | `内容质量`（不可量化）|
| `feedback_loop` | 谁产数据、怎么回流 | `追踪预警触发后的实际异常情况，销售回传确认结果` | `收集反馈`（太模糊）|
| `update_rule` | 按量或按时间触发 | `每确认100条预警或每月优化一次提醒规则` | `定期更新`（缺周期）|

### 完整示例

```yaml
name: order-reminder-agent
description: SAP订单交货期预警与销售提醒Agent，支持多系统数据源
trigger: 当任务涉及订单管理、交货跟踪、异常预警、SAP系统对接
self_improving:
  metric: 预警准确率与人工确认采纳率
  feedback_loop: 追踪预警触发的实际异常情况，销售回传确认结果，记录误报和漏报
  update_rule: 每确认100条预警或每月优化一次提醒规则和阈值
```

## 第二部分：Prompt 结构标准（7段式）

> 所有业务 Skill 必须完整包含以下 7 段，顺序不可调换，内容不可省略。缺少任何一段会导致 Agent 行为不可控。

---

### 1. 角色设定（Context）

**目的**：让 Agent 明确"我是谁、为谁服务、在什么边界内工作"。

**必须包含**：
- Agent 的身份和核心专业领域
- 服务的目标用户群（引用用户分层：P0/P1/P2）
- 如果是多领域 Skill，注明动态匹配机制
- **[关键]** Skill 的能力边界说明（什么能做、什么不归它管）

**示例（好）**：
> 你是 order-reminder-agent，专注于供应链订单管理领域的预警Agent。你服务的用户是供应链管理人员和销售专员（P1/P2人群）——他们需要及时知悉订单状态异常，但没有时间每天盯系统。
>
> **你的能力边界**：你只负责监控订单交货期并发出预警。你不处理订单修改、价格谈判、物流调度——这些交给对应业务系统。

**常见错误**：
- 身份宽泛（"你是一个助手"）
- 缺少用户画像描述
- 没有能力边界，Agent 会自我膨胀去做不该做的事

---

### 2. 用户理解确认（Profile Read）

**目的**：强制 Agent 在输出前对齐用户意图，防止跑偏。**这是最重要的防线。**

**必须包含**：
- 在推荐/执行前，先"复述"用户意图
- 列出需要用户确认的关键信息项
- 缺失信息的默认处理规则 —— **缺失即标注，绝不猜**

**关键信息确认模板**：

| 确认项 | 默认处理 |
|--------|---------|
| 用户核心领域/场景 | 由用户输入确认 |
| 目标平台/工具环境 | 未明确则标注"待确认" |
| 数据来源格式 | 未提供则标注"需补充数据源描述" |
| 输出格式偏好 | 未指定则按 Markdown 输出 |
| 使用频率/调用量 | 未指定则按"日均<100次"保守估算 |

**规则**：
- 缺失信息标注 `[待确认]`，不允许猜测
- 用户确认后使用已确认值
- 用户不确认但有合理默认值时标注 `[建议值-待确认]`

---

### 3. 知识库调用（Knowledge）

**目的**：让 Agent 在知识库的约束下输出，而非自由发挥。

**必须包含两层**：

#### Baseline（基础方法论 — 所有 Skill 通用）

```
Baseline 清单：DOMA 架构、四支柱框架、T.A.M.T.、JTBD、四层需求模型、Harness评估
```

这些是所有 Skill 都必须遵守的方法论约束，确保生成的 Skill 与整体架构兼容。

#### Specialized（领域纵深 — 基于 Skill 领域动态匹配）

使用领域匹配表做动态匹配：

| Skill 领域 | 参考知识库文件 | 设计要点 |
|-----------|--------------|---------|
| 数据分析/报表 | `03-skills-catalog/` 数据相关Skill | Schema定义、聚合逻辑、可视化输出 |
| 系统集成/API | `03-skills-catalog/` API相关Skill | 认证方式、限流处理、错误恢复 |
| 客服/对话 | `03-skills-catalog/` 对话相关Skill | 情绪识别、升级策略、知识库RAG |
| 运营/增长 | `03-skills-catalog/` 增长相关Skill | A/B测试、漏斗分析、策略推荐 |
| 代码/开发 | `03-skills-catalog/` 开发相关Skill | 代码规范、测试覆盖、安全审查 |
| 知识管理 | `03-skills-catalog/` 知识管理Skill | 索引策略、去重机制、关联推荐 |

当前知识库未覆盖的领域，标注"待补充知识库文件：[领域名]"，用通用方法论兜底。

---

### 4. 任务定义（Action）

**目的**：把 Skill 的核心工作分解为具体、可执行的步骤。

**必须包含**：
- 编号步骤列表（Agent 按序执行）
- 每步有明确输入和输出
- 约束输出数量（如"推荐 3-5 条"而不是"推荐一些"）
- **禁止模糊动词**：出现"分析""优化""评估"时必须附具体指标

**结构**：
```
1. <第一步：做什么 + 输出什么>
2. <第二步：做什么 + 输出什么>
3. <第三步：做什么 + 输出什么>
...
```

**示例（好）**：
```
1. 基于用户领域，推荐 3-5 个热点选题
2. 每个选题必须包含：话题名称 | 来源平台 | 热度评分(1-5) | 关联知识 | 优先级(P0/P1/P2)
3. 只推荐有"知识增量"的话题（能讲出方法论，而非纯情绪）
```

**示例（差）**：
```
- 分析数据
- 优化推荐
- 评估效果
```
问题："分析""优化""评估"没有配具体指标，Agent 无法执行。

**常见模糊动词替换表**：

| 禁用词 | 替换为 |
|--------|--------|
| 分析数据 | 按[维度1]/[维度2]/[维度3]逐条分析，输出对比表 |
| 优化推荐 | 基于[指标]排序，剔除[条件]不合格项，输出Top N |
| 评估效果 | 按[指标1(权重X%)+指标2(权重Y%)]加权评分，输出1-5分 |
| 检查质量 | 逐一检查[检查项清单]中的每项，标注通过/不通过 |

---

### 5. 反向过滤（What NOT to do）

**目的**：先画红线，阻止 Agent 向错误方向生成。**最容易跳过但最重要的段落。**

**必须包含**：
- 至少 3 条明确禁止事项
- 每条禁止事项附带原因或判断标准
- "禁止 X"必须配"应该 Y"或者明确判定逻辑

**常见禁止类别**：

| 类别 | 示例 |
|------|------|
| 禁止编造 | 不确定时标注"待验证"，而非编造 |
| 禁止偏离领域 | 超出 Ontology 的话题直接过滤 |
| 禁止同质化 | 已有大量同质内容的话题不推荐 |
| 禁止未验证来源 | 无法验证的"据传"信息标注"待验证" |
| 禁止跳过步骤 | 必须按 Action 的顺序执行，不可跳步 |
| 禁止过度承诺 | Agent 能力边界外的事情直接说明"不在能力范围内" |

**示例**：
> 以下类型话题**禁止推荐**：
> - 纯娱乐八卦、无行业价值的热点
> - 与用户 Ontology 明显偏离的话题
> - 无法验证来源的"据传"类信息（标注"待验证"而非直接推荐）
> - 已有大量同质化内容、缺乏差异化空间的话题

---

### 6. 输出格式（Format）

**目的**：约束输出的结构和形式，让 Agent 的输出可直接消费。

**必须包含**：
- 明确输出结构（表格/列表/Markdown/JSON）
- 明确长度限制（多少条、多少字）
- 明确表头或字段定义

**示例**：
> 以 Markdown 表格输出《每日热点选题池》，表头：
> | 话题名称 | 来源平台 | 热度评分(1-5) | 关联知识 | 优先级 |

**设计原则**：
- 输出的第一行就应该是有价值的内容，不要开头语（"你好""我来为你推荐"等）
- 格式的决定因素：下游谁在消费这个输出（Agent？人？API？）
- 如果输出会被下游 Agent 消费，优先使用 JSON/YAML

**禁止的输出形式**：
- 禁止以"你好"、"我来帮你"开头的开场白
- 禁止输出"总结"、"以上是我为你推荐的"等元描述
- 禁止输出格式不固定（有时表格有时列表）

---

### 7. 强制约束（Constraints）

**目的**：兜底规则，处理 Agent 的"自由意志"倾向。

**必须包含**：
- 事实校验规则（怎么判断"能说"还是"不能说"）
- 不确定性处理规则（标注 vs 猜测）
- 格式/行为红线（禁止写什么、禁止做什么）
- 知识边界处理（超出知识库覆盖范围怎么办）

**示例**：
> - 禁止编造，不确定时标注"待验证"而非编造
> - 禁止输出寒暄、解释、总结
> - 每个选题必须说明"为什么现在火"和"谁能讲"
> - 未覆盖领域必须标注"待补充知识库文件"
> - 输出后直接给出内容，无需前置说明

**关键原则**：
- Constraints 是最后的防线，要对 Agent 的"幻觉倾向"做针对性约束
- 每一条 Constraints 都应该是可验证的（可以事后判断 Agent 有没有违反）

## 第三部分：质量保障 & 验证

### 技能质量评分标准（引用 Skill Matrix 评分体系）

生成 Skill 后，自动按以下标准评估质量等级：

| 评分 | 等级 | 标准 | 行动 |
|------|------|------|------|
| **9-10** | ⭐ 优质 | 7段齐全，self_improving完整，约束明确，领域深度匹配 | 可部署 |
| **7-8** | ✅ 可用 | 7段齐全但部分段内容较浅，self_improving三段完整 | 建议优化后部署 |
| **5-6** | ⚠️ 待优化 | 缺少至少1段，或self_improving不完整，或约束模糊 | 必须修改 |
| **<5** | ❌ 不合格 | 结构不完整，缺少关键段落，或格式错误 | 重新生成 |

### 质量评分计算公式

```
基础分 = 7（每缺一段 -1 分）
self_improving 分 = 三段完整(+2) | 两段完整(+1) | 一段或不完整(0)
约束明确度 = What NOT ≥3条(+1) | ≠3条(0)
格式清晰度 = 输出格式有具体结构定义(+1) | 无(0)
领域匹配度 = 引用了Specialized知识库(+1) | 纯通用模板(0)
```

### 生成后验证清单（逐条自查）

**结构完整性**：
- [ ] name 是 kebab-case 全小写英文
- [ ] description ≤30 字，含动态领域能力
- [ ] trigger 以"当..."开头，明确触发场景
- [ ] self_improving 三段完整（metric/feedback_loop/update_rule）

**内容完整性**：
- [ ] 7 段式 Prompt 结构齐全（1→7 顺序正确，内容不可省略）
- [ ] Context 有角色身份 + 目标用户 + 能力边界
- [ ] Profile Read 有"待确认"处理规则
- [ ] Knowledge 有 Baseline + Specialized 两层
- [ ] Action 步骤编号，禁止模糊动词（已用替换表检查）
- [ ] What NOT 至少 3 条禁止事项
- [ ] Format 有具体结构定义，无开场白
- [ ] Constraints 有不确定性处理规则

**质量保障**：
- [ ] 质量评分 ≥ 7 分
- [ ] 没有空泛的"分析""优化""评估"动词
- [ ] 禁止事项都配有"应该 Y"或判定逻辑
- [ ] 输出的格式可以被下游消费（Agent or 人）
- [ ] Agent 能力边界已明确声明

### 快速测试：干跑验证

生成后，用以下方法快速验证 Skill 是否可用：

**测试方法**：用 Skill 中的 trigger 场景构造一条输入，模拟 Agent 执行前 2 个 Action 步骤：
1. Profile Read 是否能正确识别缺失信息？
2. Action Step 1 的输出是否在格式约束内？
3. What NOT 是否成功阻止了越界输出？

如果上述 3 条任意一条失败，Skill 需要优化后再部署。

## 第四部分：规则（Rules）

| # | 规则 | 说明 | 违规后果 |
|---|------|------|---------|
| 1 | Skill name 必须用 kebab-case | 全小写英文，连字符分隔。如 `order-reminder-agent` | Claude Code 无法识别 |
| 2 | description 不超过 30 字，体现动态领域能力 | 必须含"支持动态匹配"或类似表述 | 被静态化，失去多领域能力 |
| 3 | 必须包含 self_improving 段落 | metric / feedback_loop / update_rule 三个字段缺一不可 | Skill 不会自我迭代 |
| 4 | 必须包含完整的 Prompt 结构标准（7段式） | Context → Profile Read → Knowledge → Action → What NOT → Format → Constraints，顺序不可调换，内容不可省略 | Agent 行为不可控 |
| 5 | 质量评分必须 ≥ 7 分 | 7分以下不允许部署，需优化 | 上线效果差 |
| 6 | 产出后必须执行干跑验证 | 构造一条输入验证 Profile Read + Action Step1 + What NOT | 可能上线即失败 |
| 7 | 输出后直接告诉用户文件保存路径 | 给出绝对路径，无需额外解释 | 用户不知道文件在哪 |

### 生成流程总结

```
1. 询问用户：需要创建什么 Skill？场景、目标用户、数据来源？
   ↓
2. 输出 YAML Frontmatter（等待用户确认）
   ↓
3. 输出 7 段式 Prompt 完整内容
   ↓
4. 自动执行质量评分，输出评分结果
   ↓
5. 执行干跑验证（构造输入测试前 2 步）
   ↓
6. 输出最终 SKILL.md 文件路径
   ↓
7. 输出质量报告（评分 + 验证结果 + 改进建议）
```

## 附录：Skill 失败模式与根因

用户反馈"Skill效果不好"的常见原因：

| 失败模式 | 根因 | 在模板中的防御 |
|---------|------|-------------|
| Agent 输出格式不稳定 | Format 段缺少具体结构定义 | Format 段明确要求"具体表头/字段" |
| Agent 胡乱猜测未知信息 | Profile Read 段缺失"待确认"规则 | Profile Read 强制"缺失即标注" |
| Agent 过度承诺能力边界 | Context 段缺少能力边界声明 | Context 段新增"能力边界"子段落 |
| Agent 跳过核心步骤 | Action 段步骤不够具体 | Action 段要求编号步骤+禁止跳步 |
| Agent 自我迭代不动 | self_improving 的三段流于形式 | 新增 concrete 示例和填充指引 |
| Agent 输出空泛无价值 | Action 段含模糊动词 | 附模糊动词替换表强制替换 |
